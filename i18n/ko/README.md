# HALO Workflow v2

> [English Documentation](../../README.md)

**H**arness · **A**gentic · **L**oopback · **O**rchestration

[![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](../../LICENSE)

요구사항 분석부터 코드 리뷰까지 전체 TDD 개발 사이클을 자율적으로 실행하는 AI 에이전트 오케스트레이션 프레임워크입니다. **RTM (Requirements Traceability Matrix)** 기반 추적성과 지능형 **루프백** 복구를 제공합니다.

[Claude Code](https://docs.anthropic.com/en/docs/claude-code) 전용. **메인 에이전트**가 6개 순차 Phase를 직접 수행하고, **서브에이전트는 병렬 작업(정찰, 경쟁 설계, 리뷰)에만** 사용합니다.

> **[인터랙티브 아키텍처 다이어그램](https://FREEDOBY.github.io/halo-workflow/)**

## Quick Start

1. `.claude/` 폴더를 프로젝트에 복사
2. `/halo-workflow [기능 설명]` 실행

## Architecture

### 하이브리드 실행 모델

```
┌───────────────────────────────────────────────────────────────────┐
│  MAIN AGENT  (실행자 + 라우터 + RTM Judge)                        │
│                                                                    │
│  P1 ──→ P2 ──→ P3 ──→ P4 ──→ P5 ──→ P6 ──→ P7 ──→ P8 ──→ P9  │
│  직접    ↕      ↕     직접   직접   직접   직접    ↕     직접    │
│        정찰    경쟁                              리뷰           │
│       ┌─┴─┐  ┌─┴─┐                             ┌─┴─┐            │
│       │×2∼3│ │×2∼3│                             │ ×3 │            │
│       └─┬─┘  └─┬─┘                             └───┘            │
│     메인 Read  메인 Read+확정                                     │
│                                                                    │
│               JUDGE (P7/P8 이후)                                   │
│              ┌──── 구현 버그 → P5                                 │
│              ├──── 테스트 설계 → P6                                │
│              └──── 아키텍처 문제 → P3                              │
├────────────────────────────────────────────────────────────────────┤
│  .workflow/    체크포인트 + 상태 관리 (임시, gitignored)            │
├────────────────────────────────────────────────────────────────────┤
│  docs/ tests/ src/ reports/    제품 산출물 (영구)                   │
└────────────────────────────────────────────────────────────────────┘
```

### RTM 타임라인 (단일 진실 공급원)

```
  P1           P4           P5            P6            P7          JUDGE
  ●────────────●────────────●─────────────●─────────────●──────────▶ ◆
  │            │            │             │             │             │
  RTM 초기화   Unit TC      구현 위치      IT/E2E TC    결과 기록    판정
  REQ-ID       매핑         매핑          매핑          PASS/FAIL    근본 원인
```

## 핵심 원칙

| 원칙 | 설명 |
|------|------|
| **Main Agent First** | 순차 작업은 메인이 직접. 서브에이전트는 병렬만. |
| **제약 검증** | 외부 API/배포 환경 가정은 실제 호출로 검증 (P1). |
| **Real E2E** | E2E 테스트는 실제 환경. Mock 금지. "E2E-style" unit test 금지. |
| **RTM = 단일 진실 공급원** | P1/P4/P5/P6/P7에서 누적 업데이트. Judge가 RTM으로 판정. |
| **File = Interface** | 에이전트 간 통신과 context 복구는 파일 시스템으로만. |
| **LOOPBACK ≠ 요구사항 변경** | 요구사항은 불변. 3가지 회귀 경로 (P3/P5/P6). |

## Phases

### 메인 에이전트 직접 수행 (6 Phases)

| Phase | 역할 | 산출물 |
|-------|------|--------|
| P1 | 요구사항 분석 + 제약 검증 | `docs/requirements/` |
| P4 | Unit Test (TDD RED) | `tests/unit/` |
| P5 | 구현 (TDD GREEN) | `src/[feature]/` |
| P6 | 통합 & E2E 테스트 (실제 환경) | `tests/integration/`, `tests/e2e/` |
| P7 | 테스트 실행 + Smoke Test | RTM 결과 |
| P9 | 완료 보고서 | `reports/` |

### 서브에이전트 병렬 (3 Phases)

| Phase | 역할 | 에이전트 | 서브 역할 | 메인 역할 |
|-------|------|---------|----------|----------|
| P2 | 코드베이스 탐색 | ×2∼3 | **정찰**: 핵심 파일 목록 보고 | 보고된 파일 직접 Read |
| P3 | 아키텍처 설계 | ×2∼3 | **경쟁**: 설계안 + 참조 파일 보고 | 설계안 + 참조 코드 확인 후 확정 |
| P8 | 코드 리뷰 | ×3 | **리뷰**: 관점별 이슈 보고 | 종합 판단 |

### LOOPBACK 정책

| 트리거 | 근본 원인 | 회귀 대상 |
|--------|----------|-----------|
| P7: Unit/통합 FAIL | 구현 버그 | **→ P5** |
| P7: E2E FAIL | 테스트 설계 | **→ P6** |
| P7: 통합 FAIL (결합도) | 아키텍처 문제 | **→ P3** |
| P7: E2E에서 Mock 사용 | E2E 미달 | **→ P6** |
| P8: MAJOR | 구현 수정 필요 | **→ P5** |
| P8: CRITICAL | 아키텍처 재설계 | **→ P3** |

> **제한**: 최대 5회 LOOPBACK, Phase당 최대 2회. 초과 시 → Partial Report → P9.

## 테스트 레벨

```
Level 0: UNIT TEST        — Mock 허용, 모듈 격리 테스트 (P4/P5 TDD)
Level 1: INTEGRATION TEST — Mock 최소화, 모듈 연동 테스트 (P6/P7)
Level 2: E2E TEST         — Mock 금지, 실제 서버/브라우저/API (P6/P7)
Level 3: SMOKE TEST       — 서버 기동 + 핵심 기능 1회 확인 (P7)
```
