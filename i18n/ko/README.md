# HALO Workflow

> [English Documentation](../../README.md)

**H**arness · **A**gentic · **L**oopback · **O**rchestration

[![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](../../LICENSE)

요구사항 분석부터 코드 리뷰까지 전체 TDD 개발 사이클을 자율적으로 실행하는 3-레이어 **AI 에이전트 오케스트레이션** 프레임워크입니다. **RTM (Requirements Traceability Matrix)** 기반 추적성과 지능형 **루프백** 복구를 제공합니다.

[Claude Code](https://docs.anthropic.com/en/docs/claude-code) 전용. **하네스**가 9개의 Phase 격리 **에이전틱** 서브에이전트를 오케스트레이션하며, 테스트 실패나 리뷰 이슈 발생 시 자동으로 올바른 Phase로 회귀합니다.

> **[인터랙티브 아키텍처 다이어그램](https://FREEDOBY.github.io/halo-workflow/)**

## Quick Start

1. `.claude/` 폴더를 프로젝트에 복사
2. `/halo-workflow [기능 설명]` 실행

## Architecture

│                                                                             │
│    ┌────┐  ┌────┐  ┌────┐  ┌────┐  ┌────┐  ┌────┐  ┌────┐  ┌────┐       │

│                                                    ┌────▼────┐  ┌────┐     │
│                                                    │  JUDGE  │─→│ P9 │     │
│                                                    └────┬────┘  └────┘     │
│                          LOOPBACK ◀─────────────────────┘                   │

### RTM 타임라인 (단일 진실 공급원)

```
  P1           P4           P5            P6            P7          JUDGE
  ●────────────●────────────●─────────────●─────────────●──────────▶ ◆
  │            │            │             │             │             │
  RTM 초기화   단위 TC      구현 위치      통합/E2E TC   결과 기록    합격/불합격
  REQ-ID 생성  매핑         매핑          매핑          업데이트     판정
```

### LOOPBACK 정책

| 트리거 | 근본 원인 | 회귀 대상 |
|--------|----------|-----------|
| P7: 단위/통합 테스트 FAIL | 구현 버그 | **→ P5** (구현자) |
| P7: E2E FAIL | 테스트 설계 문제 | **→ P6** (테스트 엔지니어) |
| P7: 통합 FAIL (결합도) | 아키텍처 문제 | **→ P3** (아키텍트) |
| P8: 코드 품질 이슈 | 구현 수정 필요 | **→ P5** (구현자) |
| P8: CRITICAL 이슈 | 아키텍처 재설계 필요 | **→ P3** (아키텍트) |

> **제한**: 최대 5회 LOOPBACK, Phase당 최대 2회. 초과 시 → Partial Report → P9.

## Phases

| Phase | 에이전트 | 병렬 | 도구 | 산출물 |
|-------|---------|:----:|------|--------|
| P1 | 요구사항 분석가 | - | Read, Glob, Grep, Write | `docs/requirements/` |
| P2 | 코드 탐색기 | x3 | Read, Glob, Grep | _(내부)_ |
| P3 | 코드 아키텍트 | x3 | Read, Glob, Grep, Write | `docs/architecture/` |
| P4 | 테스트 엔지니어 (Unit) | - | Read, Write, Edit, Bash | `tests/unit/` |
| P5 | 구현자 | - | Read, Write, Edit, Bash | `src/[feature]/` |
| P6 | 테스트 엔지니어 (IT/E2E) | - | Read, Write, Edit | `tests/integration/`, `tests/e2e/` |
| P7 | 테스트 실행기 | - | Read, Bash | RTM 결과 |
| P8 | 코드 리뷰어 | x3 | Read, Glob, Grep | _(보고서 내)_ |
| JUDGE | 하네스 (RTM) | - | RTM Read | LOOPBACK 결정 |
| P9 | 보고서 작성기 | - | Read, Write | `reports/` |

## 핵심 개념

- **RTM이 단일 진실 공급원** — P1/P4/P5/P6/P7에서 누적 업데이트; Judge가 합격/불합격 판정에 활용
- **파일 = 에이전트 인터페이스** — 서브에이전트는 파일 시스템으로만 통신, 직접 통신 불가
- **LOOPBACK은 요구사항을 변경하지 않음** — 요구사항은 불변; 3가지 회귀 경로 (P3/P5/P6)
- **컨텍스트 격리** — 각 서브에이전트는 독립 컨텍스트에서 실행, Phase별 도구 권한 제한
- **TDD 우선** — P4가 실패하는 테스트 작성 (RED) → P5가 통과시킴 (GREEN) → P7이 검증
